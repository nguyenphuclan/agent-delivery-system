# Self-Detailing Requirements Protocol

## Purpose

The PO writes: *"Add sub-status, just make it work, don't care how many features it affects."* Customer says: *"Don't delay."* You can't ask 7 questions and wait 2 days for answers. You also can't guess and ship 5-10 rework cycles.

This protocol lets AI **act as both BA and PO** for the items where it has high confidence + clear source, while surfacing only the truly business-critical items for escalation.

Key shift from `requirements-clarification-protocol.md`:
- That protocol assumes "ask PO". This one assumes "PO won't engage; you must self-detail responsibly".
- Both run together. Classification decides which items go where.

## When to use

- PO description is vague / missing details
- Deadline is hard, can't wait for clarification rounds
- Project has enough existing patterns that AI can extrapolate confidently
- User explicitly invokes `--speed-mode` OR auto-triggered when PO description scores low on `requirement_clarity` heuristic

## Auto-trigger heuristic

Score requirement clarity 0-100:
- -20 if uses vague verbs ("make it work", "support", "handle")
- -15 if no acceptance criteria
- -10 per ambiguous pronoun
- -10 if PO comments contain "just", "miễn", "kệ", "doesn't matter", "ko quan trọng"
- -15 if no test cases mentioned
- -10 if scope is "+1 enum value" / "+1 column" without saying what

Score < 60 → propose self-detailing mode automatically:
> *"Requirement scores 45/100 on clarity. Many details unspecified. Run in `--self-detail` mode — AI proposes defaults, you review only critical items?"*

## Question classification (the core of this protocol)

Every question generated by `requirements-clarification-protocol.md` gets one of 3 labels:

### `needs_po` — Truly business decision (target: 1-3 items per ticket)

The answer requires PO/customer/stakeholder knowledge AI cannot infer. Examples:

- "Should sub-status be visible to customers in their portal?" — UX visibility decision
- "Should existing in-flight tasks get the new sub-status retroactively?" — data policy
- "What's the SLA for new sub-status?" — business commitment
- "Should we charge differently for tasks in new sub-status?" — pricing
- "Does this affect compliance reporting?" — regulatory

These items: **escalate, surface in Tier 2 if not addressed, block commit**.

### `ai_default` — AI can decide with high confidence (target: most items)

AI has clear source for the decision:
- Existing pattern in same codebase (3+ similar examples)
- Domain rule cited in `public-project-docs/<project>/domain/`
- Project pitfall registry recommends a specific default
- Sibling feature provides analog

AI proposes answer + source + confidence + risk-if-wrong. User reviews top-3 ranked by risk, **bulk-accepts the rest**.

Examples (for "add sub-status to ServiceTask"):
- "Transition rules for sub-status?" → AI: "Same as parent status — admin role, audit logged. Source: ServiceTaskStatusTransition.cs + 5/5 similar enums."
- "Where does the new field live?" → AI: "ServiceTask.SubStatus column, nullable, indexed if filtered. Source: pattern in 4 existing status fields."
- "Should it be in the audit log?" → AI: "Yes — every status field in this project is audited. Source: 6/6 audit_log entries cover status changes."
- "DTO mapping?" → AI: "Standard Entity↔DTO map per AutoMapper convention. Source: ServiceTaskMapper.cs."

### `gray_zone` — Could go either way (target: 1-3 items)

Defaults exist but risk-if-wrong is non-trivial. AI proposes with explicit "I'm 60% sure":
- "Default sub-status for new entities?" — propose NULL (most defensive) but flag "PO might want PENDING"
- "Should sub-status be in the default list filter?" — propose "yes, hidden by default" (UX call)
- "Reset sub-status when parent status changes?" — propose "no, decoupled" but flag

User glances, decides quickly (5-10 sec each). Default is "accept AI proposal" — 1-key.

## Speed mode UX

`do-ticket PROJECT-1234 --speed-mode` (or auto-triggered on low clarity score):

1. Run clarification questions as usual.
2. Apply classification — 80% become `ai_default`, 10% `gray_zone`, 10% `needs_po`.
3. For `ai_default`: AI writes proposed answer + source. User sees:
   - Top 3 ranked by rework-risk (PLEASE GLANCE — 30 sec total)
   - Rest in single bulk-accept prompt
4. For `gray_zone`: 1-2 quick decisions
5. For `needs_po`: surface as deferred items, will block at Tier 2 if unaddressed

**Total user time at analyze phase: ~2 minutes** (vs 15-30 min asking PO + waiting).

## Risk hedge: PR description auto-includes "AI-decided defaults"

When self-detailing was used, `pr-description.md` automatically includes:

```markdown
## AI-Decided Defaults (review if you want)

This ticket had vague requirements. AI made 12 unilateral decisions based on existing patterns. If any of these turn out wrong, here's the audit trail:

| Decision | Source | Risk if wrong | Confidence |
|----------|--------|---------------|------------|
| Enum values: PENDING_REVIEW, IN_PROGRESS, COMPLETED | ContractorStatus pattern | 3h rename | medium |
| Default value: NULL | 8/10 codebase pattern | 1h | medium |
| Indexed: yes | filterable col convention | 0.5h | high |
| ... [9 more] |

Reviewer: challenge any of these and AI will adjust.
```

This makes AI's interpretation **transparent and challengeable** — reviewer (or you) can spot wrong decisions in 2 minutes by scanning the table. If wrong, easy fix in next push.

## Defensive coding for AI-decided items

When self-detailing made many decisions, AI also tends toward defensive code:
- New columns: nullable + indexed (cheap to backfill if needed)
- New enum values: feature flag if customer-facing
- New endpoints: hidden behind feature flag for first deploy
- New events: emit but don't require consumers (additive)
- Migration: separate migration per concern (rollback-friendly)

This way, **wrong defaults are easy to fix** without re-doing the whole ticket. AI biases toward reversibility when shipping fast.

## Tier 2 modification (vs base clarification protocol)

Standard Tier 2 (in `requirements-clarification-protocol.md`) blocks if `deferred` or `dismissed` items exist. With self-detailing:

- **Block only on `needs_po` items** (1-3 items max — high signal)
- **Don't block on `ai_default` accepted items** (already a deliberate choice)
- **Don't block on `gray_zone` resolved items**

So Tier 2 reminder shrinks from "review 7 questions" to "did you really not ask PO about these 2?". Sharper signal, less noise.

## Comparison: 3 modes

| Mode | When | Items surfaced | User time | Rework risk |
|------|------|----------------|-----------|-------------|
| **Standard** (full clarification) | Normal ticket, PO available | 5-10 questions | 5-10 min review + ask PO | Low |
| **Self-detailing** (auto on low clarity) | PO vague, deadline soft | 1-3 escalate + bulk-accept rest | 2-3 min total | Low-medium (offset by defensive code) |
| **Speed mode** (`--speed-mode`) | Hard deadline, PO unavailable | 1-2 critical only | 1-2 min total | Medium (offset by defensive code + PR audit trail) |

## Anti-patterns (do NOT do)

- ❌ Classify everything as `needs_po` — defeats the protocol. Apply heuristic honestly: if 3+ similar examples exist in code, it's `ai_default`.
- ❌ Classify everything as `ai_default` — laziness in opposite direction. Genuinely business-critical things must escalate.
- ❌ Skip source citation. Every `ai_default` MUST cite a source (file path / line / pattern). Otherwise it's a guess.
- ❌ Skip the PR description audit trail. Reviewer must be able to challenge any AI decision.
- ❌ Use self-detailing for tickets where PO IS available and ticket isn't urgent. Defaulting unilaterally is debt — only take it on when worth it.
- ❌ Skip defensive coding when self-detailing was used. Easier rollback offsets unilateral risk.
