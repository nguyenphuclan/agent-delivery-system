# Risk Prioritization Protocol

## Purpose

A 40-file change presented as a flat diff is unreadable. User skims, misses the 1 line that's actually wrong, and accepts.

This protocol forces output to be **ranked by review-need**. CRITICAL items go first, capped at 5. ROUTINE items go last, easy to skip. User reads top-down, stops when comfortable, knows they covered the risky things.

## Output: `review-priorities.md`

Generated by `implement` (after main run) and by `pr-description` (for PR reviewer). Sibling of `implement-summary.md`.

```
{output}/tickets/<TICKET_ID>/review-priorities.md
```

## Section caps

Hard caps to keep review tractable:

| Section | Max items | Behavior on overflow |
|---------|-----------|----------------------|
| CRITICAL | 5 | If >5, surface top 5 by impact + flag "additional N CRITICAL items in implement-summary" |
| IMPORTANT | 10 | If >10, surface top 10 by impact + collapse rest |
| ROUTINE | unlimited | Always collapsed (file list only) |

If CRITICAL >5 → likely scope is wrong (ticket too big) → suggest splitting.

## Classification rules

Item lands in CRITICAL when ANY of:
- Decision confidence = `low` (per `decision-confidence-protocol.md`)
- Mapping flagged CRITICAL by `mapping-integrity-protocol.md`
- Migration with destructive operation (DROP, TRUNCATE, NOT NULL on existing col with no backfill)
- New unprotected endpoint (no `[Authorize]`)
- New abstraction without precedent in codebase
- Cross-service contract change (new field in event, removed field in API response)
- Auth/security surface change
- Concurrent / race condition handling code

Item lands in IMPORTANT when ANY of:
- Decision confidence = `medium`
- New endpoint (even with auth)
- New external dependency added
- Performance-sensitive code (loops over collections, query builder changes)
- Scope of change widens beyond plan.md scope
- New file > 100 LOC

Item lands in ROUTINE when:
- Confidence = `high` AND
- Follows existing pattern at `enforce`/`mandatory` priority AND
- No mapping flags AND
- Inside plan.md predicted scope

## How consuming skills use this protocol

### `implement` integration (Step 7c after green tests)

After implement-summary.md is written, generate `review-priorities.md` by:
1. Walk all decision cards from plan + implement
2. Walk all mappings from mapping-integrity scan
3. Walk file list from `git diff --name-only`
4. Classify each per rules above
5. Write review-priorities.md with caps

### `pr-description` integration

`pr-description.md` includes a "Review focus" section that's `review-priorities.md` summary (titles + 1-line each per CRITICAL/IMPORTANT). Reviewer reads PR description → knows where to focus before opening files.

### `do-ticket` plan phase integration

When showing plan to user (Q6 confidence-gated), use the same classification on plan decisions. Show CRITICAL/IMPORTANT only. ROUTINE collapsed.

## Override hooks

| Hook | Default | Override example |
|------|---------|------------------|
| `critical_cap` | 5 | hotfix: 3 (must be tiny) |
| `important_cap` | 10 | refactor: 20 (more changes expected) |
| `auto_proceed_when_no_critical` | true | Set false for paranoid mode |
| `quick_verify_required` | true | Each CRITICAL item must include "Quick verify:" actionable |

## Anti-patterns (do NOT do)

- ❌ Inflate CRITICAL section to look thorough. Cap is 5 — anything more loses signal.
- ❌ Skip "Quick verify" actionable. User must be able to verify in <2 min per item.
- ❌ Put a routine change in CRITICAL because it touches an important file. CRITICAL = decision risk, not file importance.
- ❌ Generate review-priorities.md without `implement-summary.md`. They're paired — priorities is the front door, summary is the audit trail.
- ❌ List 40 files in CRITICAL "to be safe". That defeats the protocol. Pick 5.

## Telemetry

```yaml
review_priorities:
  critical_count: 3
  important_count: 5
  routine_count: 32
  user_actions:
    - item_1_verified: true (took 90 seconds)
    - item_2_caught_bug: true (mapping wrong)
    - item_3_acked: true (correct as-is)
  user_skipped_to_routine: false
  user_overrode_classification: 0
```

`/learn` cross-references:
- If user repeatedly catches bugs in CRITICAL → calibration is good, keep
- If user never finds bugs in CRITICAL → maybe over-flagging, dial back
- If user finds bugs in IMPORTANT/ROUTINE that should've been CRITICAL → classification rules need update
